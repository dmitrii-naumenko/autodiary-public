# Методология

## Общие принципы

Autodiary использует многоуровневую систему метрик для анализа жизненных данных. Методология построена на нескольких ключевых принципах.

### Доменная организация

Метрики сгруппированы по 9 доменам: сон, активность, здоровье, тело, ментальность, семья, EQ, дневник, система. Каждый активный домен может иметь доменный скор — композитную метрику, агрегирующую ключевые показатели домена.

### Иерархия метрик

Метрики образуют дерево зависимостей. Доменные скоры (например, SleepDomainScore) вычисляются из компонентных метрик (HoursScore, OnsetScore, StabilityScore, ConsolidationScore). Компоненты в свою очередь могут зависеть от других метрик. Дерево строится автоматически из конфигурации.

### Grain (периодичность)

Каждая метрика имеет фиксированный grain — минимальный период агрегации: day, week, month или quarter. Grain определяется природой данных: пульс — дневной, усталость (FAS) — недельная, EQ — квартальный.

### Evidence-first

Каждый аналитический вывод обязан иметь доказательную базу: конкретная дата + конкретная метрика + значение. Выводы без evidence блокируются или маркируются как гипотезы.

## Правила интерпретации

### Направление метрик

Семантика направления (`better`) задаётся для каждой метрики: `high` (больше = лучше), `low` (меньше = лучше), `range` (оптимум в диапазоне), `none` (нет направления).

### Незакрытые периоды

Для текущих незавершённых периодов метрики рассчитываются по прошедшей части (effective period), а не по полной длине. Это предотвращает ложные сигналы недовыполнения.

### Статусы расчёта

- `ok` — значение рассчитано успешно
- `missing` — нет наблюдений в периоде
- `insuf_obs` — наблюдения есть, но недостаточно для расчёта (не выполнен минимум выборки)
- `no_valid_obs` — после фильтров валидности не осталось наблюдений
- `cold_start` — метрика в стадии накопления данных

### Версионирование

Каждое значение метрики хранится с `metric_version_hash` — хешем формулы расчёта. При изменении методологии устаревшие значения автоматически обнаруживаются и пересчитываются.

## Инсайты

### Критерии качества

Инсайт — это обнаруженный паттерн или вывод, прошедший проверку на доказательность. Каждый инсайт оценивается по двум осям: `confidence_score` (насколько надёжны доказательства) и `value_score` (насколько полезен для принятия решений).

### Ранжирование

```
rank = 0.35 × confidence + 0.35 × value + 0.15 × freshness(90d) + 0.15 × confirmation_bonus
```

Confirmation bonus использует логарифмическое затухание — насыщение после 5 подтверждений. Freshness затухает линейно за 90 дней.

### Жизненный цикл

`active` → `implemented` (принят и встроен в жизнь) или `dismissed` (отвергнут). Переход в финальный статус требует обязательного комментария, который становится обучающим сигналом для системы.

### Дедупликация

Новые инсайты сравниваются с существующими через cosine similarity на эмбеддингах. Порог 0.72 откалиброван на реальных данных. При совпадении — merge с обогащением доказательной базы.
